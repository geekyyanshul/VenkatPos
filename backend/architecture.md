# POS System — Architecture Document

## 1. Overview

This document describes the architecture of a single-restaurant Point-of-Sale (POS) backend built in Node.js with PostgreSQL. The system handles orders, billing, payments, and inventory across multiple outlets, with strong consistency guarantees on financial and inventory operations.

The implementation is intentionally scoped to a single backend service running against one PostgreSQL database. Distributed concerns such as offline sync, horizontal scaling, and message-broker-based event delivery are designed for in this document but not built in code, due to time constraints. Each such concern is called out explicitly in the relevant section, with the design that would be implemented next.

### 1.1 What is built

- Order CRUD with line items and per-item modifiers
- Dine-in and takeaway order types
- Strict order state machine: `CREATED → CONFIRMED → PREPARING → READY → SERVED → PAID`, with `CANCELLED` reachable from `CREATED` or `CONFIRMED`
- Per-outlet menu and configurable tax rate (multi-tenancy)
- Inventory deduction on order confirmation, with both pessimistic locking and database-level constraints
- Bill generation with discounts (flat or percentage) and tax computation
- Payment recording with multiple payment methods (CASH, UPI, CARD)
- Idempotency keys on payment to make the API safe to retry
- An in-process event bus (`order.confirmed`, `order.status_changed`, `payment.recorded`) that decouples downstream side effects
- Database-level guards for double billing (UNIQUE constraint on `bills.order_id`) and overselling (`CHECK (stock >= 0)`)

### 1.2 What is designed but not built

- Offline-capable clients with sync on reconnection
- Split payments across multiple methods on a single bill
- Idempotency keys on order creation
- Horizontal scaling to 10,000 orders/hour
- Message-broker-based event delivery (Kafka/RabbitMQ)

The Express server is stateless: every request opens a transaction against PostgreSQL, completes its work, and commits or rolls back. No request-level state is held in memory. This is what makes the service horizontally scalable in principle (see $10).

## 2. Data Model

### 2.1 Tables

The full DDL is in `schema.sql`. Summary of each table and the design choices behind it:

**`outlets`** — One row per restaurant. Multi-tenancy is implemented column-wise: every other table that contains tenant data references `outlet_id`. Each outlet has its own configurable tax rate.

**`menu_items`** — Dishes available at an outlet. Each item belongs to exactly one outlet. There is no cross-outlet menu sharing.

**`inventory`** — Stock per menu item. The `CHECK (stock >= 0)` constraint is the database-level guarantee against overselling: even if every concurrency mechanism in the application code fails, the database itself will refuse to let stock go negative.

**`orders`** — One row per customer order. Three design choices worth highlighting:
- The primary key is a **UUID** generated by the application, not a database-generated integer. This is deliberate. UUIDs allow IDs to be generated client-side (including offline), without coordinating with a central server. This is a foundation for offline-first clients (see §9).
- The `status` column has a `CHECK` constraint listing only the seven valid state values. This is a database-level guard against ever storing an invalid state.
- The `version` integer column is used for optimistic locking on concurrent edits (see §3).

**`order_items`** — Line items in an order. The `unit_price` column is a **snapshot** of the menu item's price at the time the order was placed. Bills are computed against snapshotted prices, never against current menu prices. This is essential for correctness: a price change tomorrow must not retroactively change yesterday's bill.

**`bills`** — One bill per order. The `UNIQUE` constraint on `bills.order_id` is the database-level guarantee against double billing. Two concurrent requests to generate a bill for the same order will both attempt to insert; the second will fail with a uniqueness violation. The application catches this and returns a 409 Conflict response.

**`payments`** — One or more payment rows per bill. Currently each bill has exactly one payment in the implementation, but the schema supports split payments across multiple methods.

**`idempotency_keys`** — A cache of processed requests. Each row stores the response that was returned for a given key. When a request arrives with a key that has been seen before, the cached response is returned without re-executing the operation.

### 2.2 Why these data-level guarantees matter?

The design follows the principle of **defense in depth**: every important invariant (no negative stock, no double bills, only valid order statuses, only valid payment methods) is enforced at *both* the application layer and the database layer. The application layer protects the happy path; the database layer protects against application bugs, race conditions the code didn't catch, and any direct database modification.

This means a database administrator running a stray UPDATE statement, a buggy code path, or a concurrent race condition all hit the same wall: the data layer refuses to accept invalid state.

## 3. Consistency & Concurrency Model

### 3.1 Consistency model

The system is not uniformly strongly consistent or eventually consistent. Different operations have different consistency requirements, and the design picks the right model for each:

| Operation | Consistency required | Why |
|---|---|---|
| Inventory deduction on order confirmation | Strong | Overselling has real-world cost (customer told item is available, kitchen has none). |
| Payment recording | Strong | Double-charging is unrecoverable trust damage. |
| Bill generation | Strong | Double bills create financial reconciliation problems and customer complaints. |
| Order state transitions | Strong | An order in two states at once breaks every downstream assumption. |
| Menu reads | Eventual (acceptable) | Slight staleness on menu data is harmless. |
| Order history reads | Eventual (acceptable) | Reading a list of past orders does not need to reflect this-second writes. |

In the current single-database implementation, all reads see the latest committed write because they hit the same PostgreSQL instance. The eventual-consistency tolerance becomes meaningful only when read replicas or caches are introduced for scaling (see §10).

### 3.2 Transactional boundaries

Every mutating operation runs inside a PostgreSQL transaction (`BEGIN ... COMMIT`). The transactional boundaries are:

- **Create order**: insert into `orders` + insert into `order_items` for each line item. All-or-nothing.
- **Transition order**: read current state with version → validate transition → if confirming, deduct inventory rows → update order with version increment. All-or-nothing.
- **Generate bill**: read order + items → compute totals → insert bill. All-or-nothing.
- **Record payment**: check idempotency key → lock bill → insert payment → update bill status → update order status → cache the response under the idempotency key. All-or-nothing.

If anything fails inside a transaction — a constraint violation, a network error, an exception thrown by the code — the entire transaction is rolled back. This is the foundation of every consistency guarantee in the system.

### 3.3 Concurrency strategies

The system uses **both** pessimistic and optimistic locking, deliberately, in different places:

#### Pessimistic locking (SELECT … FOR UPDATE)

Used where conflicts are common, locks are held briefly, and correctness is non-negotiable. The classic case: two concurrent confirmations for the last unit of an item.

**Inventory deduction** acquires a row-level lock on each `inventory` row before reading the stock value:

```sql
SELECT stock FROM inventory WHERE menu_item_id = $1 FOR UPDATE;
```

Concurrent transactions hitting the same row block until the first transaction commits or rolls back. This serializes inventory access for the same item, while leaving other items unaffected.

**Payment recording** acquires a row-level lock on the `bills` row:

```sql
SELECT bill_id, total, status FROM bills WHERE bill_id = $1 FOR UPDATE;
```

Combined with the bill's `status` field check (a paid bill cannot be paid again), this prevents double payment even if two requests arrive simultaneously without an idempotency key.

#### Optimistic locking (version column)

Used where conflicts are rare, where blocking would harm throughput, and where "your view is stale, please refresh" is an acceptable user experience. The classic case: two waiters editing the same order from different tablets.

The `orders` table has a `version` integer column. Every update reads the current version, then writes back with `WHERE version = <expected>` and `SET version = version + 1`. If another transaction updated the row in the meantime, the WHERE clause matches zero rows, and the application detects this by checking `rowCount`:

```sql
UPDATE orders
   SET status = $1, version = version + 1, updated_at = NOW()
 WHERE order_id = $2 AND version = $3;
```

If `rowCount = 0`, the application throws a `ConflictError` (HTTP 409). The client can retry by re-reading the order to get the new state and reapplying the change.

#### Why not lock everything pessimistically

Pessimistic locking is correct but throughput-expensive. Under high load, every concurrent edit on the same row blocks. Optimistic locking lets the database handle thousands of non-conflicting updates in parallel and only spends time on conflict resolution when there *is* a conflict. The trade-off: with optimistic locking, the conflict-resolution code lives in the application; with pessimistic locking, it lives implicitly in the database's lock manager.

### 3.4 The race conditions this design defends against

For each scenario, the defense:

| Scenario | Defense |
|---|---|
| Two waiters confirm orders for the last unit of an item | Pessimistic lock on inventory row + `CHECK (stock >= 0)` constraint |
| Two requests try to generate a bill for the same order | `UNIQUE(order_id)` constraint on `bills` table |
| Two requests pay the same bill | Pessimistic lock on bill row + status check (`UNPAID → PAID`) + idempotency key cache |
| Two waiters edit the same order from different tablets | Optimistic locking via `version` column on `orders` |
| Network retry of a payment that already succeeded | Idempotency key returns the original response (see §4) |
| Kitchen marks order PREPARING at the same moment customer cancels it | Optimistic locking ensures only one transition wins; the other gets a 409 |

## 4. Idempotency & Fault Tolerance

### 4.1 Why idempotency matters in a POS

Networks fail in subtle ways. A payment request might:
- Reach the server, succeed, and the response gets lost on the way back to the tablet
- Get retried by the tablet (which has no idea the original succeeded)
- Be double-tapped by a frustrated customer who didn't see a confirmation

In all three cases, the server receives the same payment request twice. Without idempotency, the customer is charged twice. With idempotency, the second request returns the original response without re-charging.

The assignment explicitly requires "APIs must be safe to retry" and "no duplication of payments." Idempotency is how this requirement is met.

### 4.2 The idempotency-key pattern

For every mutating request that must be safe to retry, the client generates a unique key and sends it in an HTTP header:

POST /payments
Idempotency-Key: pay-7f3a9c-xyz
{ "bill_id": "...", "amount": 504, "method": "UPI" }

The server, before doing any work:

1. Looks up the key in the `idempotency_keys` table
2. If found: return the cached response status and body. No new work is done.
3. If not found: do the work, store the (key, response) pair, and return the response.

The implementation lives in `paymentService.js`. The key lookup, the actual payment processing, and the cache write all happen inside a single database transaction so that the lookup-then-write sequence is atomic — two concurrent requests with the same key cannot both find "no entry" and both proceed.

### 4.3 Defense in depth on payment

Idempotency keys are the request-layer defense. The data layer adds two more:

1. **Pessimistic lock on the bill row** — prevents two concurrent payments to the same bill from racing.
2. **Bill status check** — a bill in status `PAID` cannot accept a new payment; the application returns 409 Conflict.

This means: even if a client somehow sends two payment requests with *different* idempotency keys to the same bill (e.g., generating a fresh key on each retry, which would be a client bug), the bill status guard catches it.

The discussion-round answer to "how do you prevent double billing?" is exactly this layering: idempotency at the request layer, pessimistic locking at the transaction layer, status guard at the data layer, and the `UNIQUE(order_id)` constraint at the schema layer. Four independent defenses.

### 4.4 What is currently NOT idempotent

In the current implementation, only **payment** has idempotency keys. Order creation does not. This is a known gap. The same pattern applies trivially: client generates a key, server checks the cache, server stores the response. The work is roughly 30 minutes of code that was scoped out due to time. The implementation pattern is identical to payment.

State transitions are *naturally* idempotent in a different way: transitioning an order from `CONFIRMED → PREPARING` succeeds the first time. The second call from the same state fails the state-machine check (you cannot transition from `PREPARING → PREPARING`) and returns a 400. From the client's perspective, this is safe — calling the same transition twice does not create duplicate effects, it just returns an error on the second call. A stricter idempotent design would return a 200 success on the duplicate call. The current behavior is an acceptable middle ground.

### 4.5 Fault tolerance summary

Failures the system handles correctly:

| Failure | Outcome |
|---|---|
| Server crash mid-transaction | Database rolls back automatically. Partial state never visible. |
| Network failure between client and server | Client retries with same idempotency key. Cached response returned. No duplicate work. |
| Network failure between server and database | Transaction times out, rolls back. Client sees error, retries safely. |
| Database constraint violation (e.g., negative stock) | Transaction rolls back. Application returns a 400 with a clear error message. |
| Concurrent state transition by two clients | Optimistic lock detects the conflict; one succeeds, the other gets 409. |
| Two concurrent inventory deductions for the last item | Pessimistic lock serializes them; first succeeds, second sees stock=0 and gets 400. |
| Server crash after commit, before sending response | Client retries with same idempotency key. Cached response returned (the work succeeded; the cache was written inside the same transaction). |

The last case deserves emphasis. Because the idempotency-key cache write is part of the *same transaction* as the actual work, the cache is written if and only if the work was committed. There is no window where the work succeeded but the cache says "not yet processed" or vice versa.

## 5. State Machine

### 5.1 Order lifecycle states

An order moves through one of two paths:

**Happy path:** `CREATED → CONFIRMED → PREPARING → READY → SERVED → PAID`

**Cancellation path:** `CREATED → CANCELLED` or `CONFIRMED → CANCELLED`

Once an order reaches `PAID` or `CANCELLED`, it is terminal: no further transitions are allowed.

### 5.2 Allowed transition map

The state machine is defined as an explicit data structure in `orderService.js`:

```javascript
const ALLOWED_TRANSITIONS = {
  CREATED:   ['CONFIRMED', 'CANCELLED'],
  CONFIRMED: ['PREPARING', 'CANCELLED'],
  PREPARING: ['READY'],
  READY:     ['SERVED'],
  SERVED:    ['PAID'],
  PAID:      [],
  CANCELLED: []
};
```

Every transition request goes through a `canTransition(from, to)` check before the database is touched. Invalid transitions are rejected with a 400 response naming both states explicitly, e.g. `"cannot transition from CONFIRMED to PAID"`.

### 5.3 Why the state machine is a first-class concept

In a naive implementation, each endpoint that changes order status (kitchen marks PREPARING, customer marks PAID, etc.) would have its own status-update logic, and the rules would be scattered across the codebase. This invites bugs where one endpoint allows a transition another endpoint forbids.

In this design, **every state change goes through one function** (`transitionOrder`) and one map (`ALLOWED_TRANSITIONS`). Adding a new state or a new transition is a one-line change. Reading the rules takes one glance at the map.

### 5.4 Defense in depth on state

Three layers protect against an order ever holding an invalid status:

1. **Application-layer transition check** — the `canTransition` function blocks invalid moves before any SQL runs.
2. **Optimistic locking via `version`** — even if two clients each pass the transition check independently, only one's UPDATE matches the version; the other gets a 409 ConflictError.
3. **Database-level CHECK constraint** — the `orders.status` column has `CHECK (status IN ('CREATED','CONFIRMED','PREPARING','READY','SERVED','PAID','CANCELLED'))`. A bug, a misuse of the database, or any other path that tries to write a value outside this set is rejected by PostgreSQL itself.

### 5.5 Side effects tied to specific transitions

Some transitions have side effects that must be atomic with the state change:

- **`CREATED → CONFIRMED`**: inventory is deducted for every line item. The deduction happens in the same transaction as the status change. If inventory is insufficient for any item, the transaction rolls back: status stays at CREATED, inventory is unchanged. There is no half-confirmed state.
- **`SERVED → PAID`**: triggered by payment success, not as a direct status PATCH. The payment service updates the order status as part of its transaction.

Other transitions (PREPARING, READY, SERVED) are pure status updates with no side effects beyond the version bump and the emitted event.

### 5.6 Why no automatic refund on cancellation

The current implementation allows `CANCELLED` from `CREATED` (no inventory was deducted yet) and from `CONFIRMED` (inventory was deducted). The `CONFIRMED → CANCELLED` path *does not* automatically restore inventory in this implementation.

This is a deliberate scope choice, not a bug. In a real restaurant, cancelling a confirmed order is a manual operation: the kitchen may have started prep, ingredients may have been consumed, and the question of whether to restore inventory is a business decision. A real implementation would either:
- Restore inventory atomically with the cancellation, accepting that this can produce overcounting if ingredients were used; or
- Leave inventory deducted and require a separate "restock" operation by a manager.

The chosen path here is the second: cancellation is recorded, inventory is not auto-restored, and a future enhancement would add a manager-initiated restock endpoint.

## 6. Event-Driven Design

### 6.1 The pattern

Order workflows have side effects: confirming an order should notify the kitchen, deduct inventory, log analytics, and possibly send the customer an SMS. If all of these were called directly from `transitionOrder`, the order code would be tightly coupled to every concern in the system, and a slow notification service would slow down every order confirmation.

The system uses an **event bus** (`src/events.js`) — a Node.js `EventEmitter` instance — to decouple state changes from their side effects:

```javascript
eventBus.emit('order.confirmed', { order_id });
```

Listeners subscribe independently:

```javascript
eventBus.on('order.confirmed', (data) => {
  // do something on confirmation
});
```

The order workflow does not know who is listening or what they do. New listeners can be added without modifying the order code.

### 6.2 Events emitted by this system

- `order.status_changed` — fired on every order state transition, with `{ order_id, from, to }`
- `order.confirmed` — fired specifically when a transition lands on `CONFIRMED`. This is a more semantic event for downstream consumers that only care about confirmation, not all transitions.
- `payment.recorded` — fired when a payment is successfully recorded.

### 6.3 Events fire after commit

Events are emitted *after* the database transaction commits, never inside it. This is important: a listener that queries the database expects to see the committed state. Emitting an event mid-transaction would let listeners read stale or unwritten data.

The current listeners are logging-only (they print to stdout for visibility during testing). In a production version, the same events would trigger:

- Pushing the new order to a kitchen display system
- Sending an SMS via Twilio when an order moves to READY
- Writing to an analytics warehouse
- Updating loyalty points after payment

### 6.4 In-process now, broker-backed later

The current implementation uses an in-process event emitter. This works because the entire system runs inside a single Node.js process. It has known limitations:

- **No durability** — if the server crashes between emitting an event and the listener processing it, the event is lost.
- **No cross-process delivery** — listeners must run in the same Node.js process as the emitter.
- **No replay** — there's no way to replay past events to a new listener.

In production, the in-process bus would be replaced by **Kafka or RabbitMQ**:

- Each event becomes a message on a topic (`orders.status_changed`, `payments.recorded`).
- Listeners become independent consumer services.
- Messages are durable: a crashed consumer resumes from where it left off when restarted.
- The event log can be replayed to bootstrap new consumers (e.g. a new analytics pipeline).

The application code shape stays the same. `eventBus.emit()` becomes `kafkaProducer.send()`, and listener registration becomes a Kafka consumer. This is a deliberately portable design — events are emitted in a way that maps cleanly to a real broker without rewriting the business logic.

## 7. Multi-Tenancy

### 7.1 Approach

The system supports multiple restaurants (outlets) on a single database instance using **shared-database, shared-schema multi-tenancy**: every tenant-owned row carries an `outlet_id` foreign key, and queries scope to the appropriate outlet.

There are three common multi-tenancy models, each with different trade-offs:

| Model | Description | Trade-offs |
|---|---|---|
| **Database per tenant** | Each tenant gets its own database | Strongest isolation, expensive to operate at scale, hard to do cross-tenant analytics |
| **Schema per tenant** | One database, separate schemas | Medium isolation, manageable up to ~hundreds of tenants |
| **Shared schema with `tenant_id` column** | One database, one schema, every row tagged | Cheapest to operate, simplest to query, weakest isolation, scales to thousands of tenants |

This implementation uses the third model. For a POS system targeted at small-to-medium restaurants, shared schema is the right default: it's the cheapest to operate, the simplest to scale to thousands of outlets, and the easiest to do cross-outlet analytics on.

### 7.2 What multi-tenancy buys

- **Independent menus**: each outlet's `menu_items` are scoped by `outlet_id`. An outlet's menu cannot be read or modified by code that doesn't know the outlet ID.
- **Independent tax rates**: `outlets.tax_rate` is per-outlet. Bills are computed using the tax rate of the outlet the order belongs to.
- **No cross-tenant leakage in normal queries**: every relevant query includes the outlet ID as part of its WHERE clause or JOIN condition.

### 7.3 What is NOT yet enforced

Two important hardenings that are designed but not implemented:

1. **Authentication and authorization**: there is no auth layer in the current implementation. In production, the API would authenticate the caller (typically via a JWT containing the user's outlet_id) and reject any request whose outlet_id parameter does not match the authenticated outlet. Currently any caller can pass any outlet_id.

2. **PostgreSQL Row-Level Security (RLS)**: a defense-in-depth layer where the database itself rejects queries that try to read rows from a different tenant, even if the application layer has a bug. RLS would be enabled with policies like `USING (outlet_id = current_setting('app.outlet_id')::int)` on every tenant table. The application would set `app.outlet_id` at the start of each request based on the authenticated user.

## 8. Failure Scenarios

This section walks through the failures that real POS systems encounter and how this design responds to each. Every scenario is one a senior reviewer is likely to ask about!!

### 8.1 Server crashes mid-transaction

**Scenario**: An order is being confirmed. The server has read the inventory rows, locked them, decremented stock, and inserted the order — but crashes before COMMIT.

**Outcome**: PostgreSQL rolls the transaction back(ROLLBACK function) automatically when the connection is dropped. The locked rows are released. No partial state is visible to anyone. The client sees a connection error and can retry safely (the work didn't happen).

This is the foundational guarantee of the transactional design. Atomicity (the A in ACID) is what protects against this failure class.

### 8.2 Network failure between client and server

**Scenario**: The tablet sends a payment request. The server processes it successfully and starts to return a 200 response. The network drops before the response reaches the tablet. The tablet does not know whether the payment succeeded.

**Outcome**: The tablet retries with the **same idempotency key**. The server looks up the key, finds the cached response from the original successful transaction, and returns it. No second payment is recorded.

This is precisely the case the idempotency-key pattern is designed for. Without it, the customer would be charged twice on every dropped response.

### 8.3 Network failure between server and database

**Scenario**: The server is mid-transaction. The connection to PostgreSQL drops.

**Outcome**: The connection's outstanding queries fail. The transaction is implicitly rolled back on the database side when the connection times out. The server's transaction wrapper catches the error, calls ROLLBACK (which is a no-op on a dead connection but harmless), and returns a 500 to the client. The client retries with the same idempotency key (for payments) or a fresh request (for other operations). No partial state is visible.

### 8.4 Concurrent writes to the same row (race condition)

**Scenario A — last item in stock**: Two waiters confirm orders for the last unit of an item at the same instant.

**Outcome**: The first transaction acquires the `FOR UPDATE` lock on the inventory row, decrements stock to 0, commits. The second transaction was blocking on the lock; it unblocks, reads the new value (0), sees insufficient stock, throws a ValidationError, and the transaction rolls back. The second customer is told "out of stock." The first customer's order is confirmed normally.

**Scenario B — two waiters editing the same order**: Both tablets read order version 5. Tablet A adds an item, writes back with `version = 6 WHERE version = 5`. Tablet B adds a different item, writes back with `version = 6 WHERE version = 5`.

**Outcome**: A's UPDATE matches the row and increments to version 6. B's UPDATE matches zero rows (the version is no longer 5). The application detects `rowCount = 0`, throws ConflictError, returns 409 to B's client. B's tablet refreshes the order and retries.

### 8.5 Database constraint violation

**Scenario**: A bug in the application code or a malicious input attempts to write data that violates a check constraint — say, inserting a payment with an invalid method, or somehow attempting to set negative stock.

**Outcome**: PostgreSQL rejects the write. The transaction rolls back. The application catches the error and returns a 400 Bad Request with a meaningful message. The constraint at the database level is the last line of defense — the application code should reject these inputs earlier, but if it fails to, the database does not.

### 8.6 Idempotency key collision

**Scenario**: A client sends the same idempotency key with *different* request bodies. (This would be a client bug.)

**Outcome — current implementation**: The cached response is returned. The new request body is ignored. This is permissive but not strictly safe.

**Hardening that would be added in production**: store a hash of the request body alongside the response. On a key collision, compare hashes. If the body differs, return 409 Conflict. This is what production payment processors (e.g. Stripe) do.

### 8.7 Partial system outage (network partition)

**Scenario**: In a multi-server deployment, the database is reachable from server A but not from server B due to a network partition.

**Outcome — single-database setup as currently built**: server B's requests fail. The system is unavailable from the affected server, but the database remains a single source of truth, so no inconsistency develops. CAP-wise, this is a CP system: it sacrifices availability during partition to preserve consistency.

**For multi-database / multi-region setups**, consistency vs. availability trade-offs become explicit. This system would still favor consistency: a payment that cannot reach the primary database should fail and surface to the user, not be queued for later application against potentially-stale state.

### 8.8 Order created offline (current limitation)

**Scenario**: A tablet creates an order while the network is down.

**Outcome — current implementation**: The request fails. The tablet has no local persistence. The order is lost.

**Designed (not implemented)**: see §9. The design is for the tablet to write to a local store, queue the request in an outbox, and sync on reconnection.

### 8.9 What this system explicitly does NOT do

It is honest to call out what the design does not solve:

- **Byzantine failures** (malicious actors inside the trust boundary): not in scope.
- **Database total loss without backup**: relies on standard PostgreSQL replication and backup, which are operational concerns outside this design.
- **Multi-region active-active**: the current design is single-region. Multi-region active-active is a significantly more complex problem (requires careful conflict resolution and is closer to the offline-sync problem in §9).


## 9. Offline Sync Design

This section describes the design for offline-capable clients. **It is not implemented.** Building offline sync correctly is a 1–2 week effort with significant test surface, well outside the time budget for this assignment. The design below is what would be implemented next.

### 9.1 The problem

A POS tablet must keep working when the network drops. Waiters must be able to take orders, confirm orders, generate bills, and accept payments while offline. When connectivity returns, the local state must reconcile with the server.

This is genuinely hard because while the tablet was offline:

- Other tablets may have changed shared resources (inventory).
- The server's view of the world has moved on.
- Two tablets may have made conflicting changes to the same order.

A naive sync ("send everything I did, server applies it") loses data, double-deducts inventory, or silently overwrites another tablet's work.

### 9.2 Design principles

1. **Client-generated IDs.** Every entity (orders, bills, payments) gets its UUID at creation time on the tablet. No server round-trip needed to mint IDs. This is why `orders.order_id` is a UUID, not a serial integer — the schema is already offline-ready in this respect.

2. **Append-only outbox.** Every mutating action the tablet performs is recorded as an event in a local outbox before any UI confirmation. The outbox is durable (SQLite or IndexedDB on the tablet). When online, events flush to the server in order.

3. **Domain-aware conflict resolution.** Generic "last-write-wins" loses too much data. Resolution rules are per-entity-type, encoding the business meaning of the operation.

4. **Idempotent sync.** Every event in the outbox carries an idempotency key. Re-syncing the same event is a no-op on the server. Tablets that lose connectivity mid-sync can safely retry from the start of the unsynced batch.

### 9.3 Local data model on the tablet

The tablet maintains a local replica of:

- Its own outlet's `menu_items` and `inventory` (read-only, refreshed periodically)
- Orders the tablet has created or is currently editing (read-write)
- An `outbox` table of pending events, with timestamps and idempotency keys

### 9.4 The sync protocol

When the tablet detects connectivity:

**Step 1 — Push outbox events to server.**
The tablet sends pending events in order:

POST /sync/events
[
{ type: 'order.created', order_id: '...', idempotency_key: '...', payload: {...} },
{ type: 'order.confirmed', order_id: '...', idempotency_key: '...' },
{ type: 'payment.recorded', bill_id: '...', idempotency_key: '...', payload: {...} }
]

The server processes each event:

- If the idempotency key has been seen, it's a duplicate from a previous half-sync. Skip.
- Otherwise, apply the event using the same business logic as live API calls (transactions, locking, state machine, inventory checks).
- Each event either succeeds, fails (returns an error to the tablet), or partially fails (some events succeed, later ones fail because the world changed).

**Step 2 — Reconcile failures.**
For each failed event, the server returns a structured failure reason. The tablet handles each by domain rule (next section).

**Step 3 — Pull server changes.**
The tablet fetches anything it might have missed: menu changes, inventory levels for items the tablet shows, status changes on orders the tablet doesn't own. This is straightforward last-pull-timestamp polling.

### 9.5 Conflict resolution rules, by entity type

| Conflict | Resolution |
|---|---|
| Two tablets created different orders for the same table | Both succeed. They have different UUIDs, so no collision. The two orders coexist; staff resolves ambiguity by talking to each other. |
| Offline tablet confirmed an order that, by sync time, has insufficient inventory | The confirmation event fails server-side. The tablet shows the order as "needs reconciliation": the manager sees the situation and decides — apologize and refund, substitute, or accept stockout. |
| Two tablets edited the same order while offline | Compare per-field. If they touched different fields (one added an item, one changed table number), merge. If they touched the same field, surface to the user with both versions visible. |
| Offline tablet recorded a payment for a bill that was paid online in the meantime | Idempotency key check at the server prevents double payment. The tablet's payment event is rejected with "bill already paid." Tablet shows reconciliation prompt to the operator. |
| Offline tablet's order was cancelled by another tablet | The status transition CREATED → CONFIRMED fails on sync (state machine: cannot transition from CANCELLED). Tablet shows "order was cancelled by another device" prompt. |

The principle: **automatic resolution where the rules are unambiguous, manual reconciliation surfaced to humans where they aren't.** Silently picking a winner is worse than asking.

### 9.6 What about CRDTs?

Conflict-free Replicated Data Types are the academically-elegant solution to multi-writer sync without a central coordinator. They are used in production by Figma, Riak, and others.

For a POS, CRDTs are overkill in most cases:

- The interesting conflicts (inventory, payments) are not naturally CRDT-able. You cannot make money "addition-only" or "set-merging" — overspending must be detected and rejected, not silently merged.
- The cases where CRDTs would help (collaborative editing of an order's notes field) are rare and not worth the complexity.

A pragmatic POS uses **append-only event logs** for the operations that matter (every action becomes an immutable event with an idempotency key) and **last-write-wins with manual reconciliation for surfacing conflicts** for the rest. This is the design above. CRDTs are not necessary.

### 9.7 Offline duration limits

The longer a tablet is offline, the staler its view of the world. Two practical limits:

1. **Inventory accuracy degrades.** A tablet offline for an hour may have confirmed orders against stock that has long since run out. The reconciliation flow handles this, but at some point it becomes unworkable. A reasonable policy is to lock the tablet to read-only mode after, say, 4 hours offline.

2. **Menu drift.** Prices may have changed. Items may have been removed. The tablet should warn that menu data is stale beyond some threshold and refuse to take orders until refreshed.

These limits are policy, not infrastructure. The system supports operating offline indefinitely; the question is whether we *let* it.

### 9.8 Why this is not built

Implementing offline sync requires:

- A local database (SQLite or IndexedDB) on every client
- A reliable outbox with retry logic
- A sync endpoint on the server with batched event processing
- Per-event reconciliation logic
- A UI flow for surfacing manual reconciliation needs
- A test harness that simulates network partitions, partial syncs, conflicting events

This is a significant subsystem in its own right — production POS systems like Toast and Square have engineering teams dedicated to it. It was scoped out of this submission in favor of a solid online-only core plus this design document.

The schema is already offline-friendly: UUIDs as primary keys, idempotency-key infrastructure in place. Adding offline sync is incremental work, not a rewrite.

## 10. Scaling Strategy to 10,000 Orders/Hour

10,000 orders per hour is roughly **3 orders per second** sustained, with peak bursts of 10–20/second during dinner rush. This is a moderate load — well within the reach of a properly-tuned single-server system, and easily handled by a small horizontal cluster.

This section walks through how the current system behaves at scale, where it would fail first, and what would be done to push past each ceiling.

### 10.1 Current capacity envelope

The current implementation is a single Node.js process talking to a single PostgreSQL instance over a connection pool of 20 connections.

Back-of-envelope:

- Each order goes through ~6 database transactions across its lifecycle (create, confirm, three state changes, bill, payment).
- Each transaction is short — a handful of queries, no long-held locks, sub-100ms typical latency.
- A single PostgreSQL instance on modest hardware comfortably handles 1,000–5,000 transactions per second on this kind of workload.
- 10,000 orders/hour ≈ 17 transactions per second. Two orders of magnitude below the database's headroom.

So the bare numbers say the current design handles 10k/hour without architectural changes. The interesting questions are: where does it actually start to break, and what do we do then?

### 10.2 Where it fails first

In order of likely failure under load:

**1. Connection pool exhaustion.**
The `pg` pool defaults to 20 connections. Each in-flight request holds a connection for the duration of its transaction. If transactions slow down (e.g., a long-held inventory lock during a rush), the pool fills, and incoming requests queue waiting for a connection. Eventually, requests time out.

*Fix*: increase pool size, but PostgreSQL has its own per-server connection limits. Beyond ~100–200 connections per server, performance degrades. The real fix at higher loads is **PgBouncer** or **pgcat** — connection-poolers that multiplex thousands of application-level connections onto a small number of actual database connections. Industry standard for Postgres at scale.

**2. Hot row contention on inventory.**
The `FOR UPDATE` lock on inventory rows serializes concurrent confirmations of the same item. If 50 tablets all try to order the most popular item at peak, they form a queue. Each transaction is short, so the queue moves quickly — but at very high contention, p99 latencies spike.

*Fix*: for items where contention is real (high-volume staples), denormalize stock into a counter that can be decremented atomically without a row lock (`UPDATE … SET stock = stock - 1 WHERE menu_item_id = X AND stock >= 1`). The CHECK constraint still backstops correctness. This avoids the explicit `FOR UPDATE` and lets PostgreSQL handle concurrency through MVCC.

**3. Single Node.js process.**
Node is single-threaded. CPU-bound work (none in this system, but logging and JSON parsing add up) can saturate one core under load.

*Fix*: run multiple Node processes behind a load balancer (nginx, AWS ALB). They're stateless; no request affinity needed. This is horizontal scaling and is essentially free with this design.

**4. Idempotency key table growth.**
Every payment writes a row to `idempotency_keys`. Over a year at 10k orders/hour, that's ~88 million rows. The table grows indefinitely and lookups slow as the index gets bigger.

*Fix*: add a TTL. Idempotency keys are only useful for retry windows of seconds-to-minutes. A scheduled job deletes keys older than 24 hours. Or move the cache to **Redis** with a 24-hour expiry — much faster, no PostgreSQL load, no growth.

**5. Read pressure on menu/inventory.**
Every order creation reads menu items. Every order page-load reads order data. Read-heavy workloads start showing up as primary database load.

*Fix*: PostgreSQL read replicas for read-heavy queries (menu reads in particular). Writes still go to the primary; reads of slow-changing data can tolerate replication lag. Menu and outlet data is also a strong fit for a Redis cache with a few-minute TTL.

### 10.3 The scaling staircase

Here's the rough order in which scaling work happens as load grows:

| Load | Architecture | Bottleneck |
|---|---|---|
| Up to ~5k/hour | Single Node, single PG | Whatever surfaces first; usually fine |
| 5k–50k/hour | Multiple Node processes behind LB, single PG | Connection limit on PG |
| 50k–500k/hour | + PgBouncer, + read replicas, + Redis cache | Write throughput on primary PG |
| 500k+/hour | + Sharding by `outlet_id`, + Kafka for events | Single primary writes, cross-tenant queries |

The current system sits comfortably at the top of the first row. The transition to the second row is one afternoon of ops work. The third row is a project, but a known-shape project. The fourth row is where serious engineering happens — and is also where Toast and Square actually live.

### 10.4 What stays the same as load grows

The design decisions that make scaling cheap:

- **Stateless application servers.** No in-memory session state. Any server can handle any request. Horizontal scaling is `add another box behind the load balancer`.
- **Per-tenant scoping.** All tenant data is keyed by `outlet_id`. Sharding by outlet (when the time comes) is straightforward — outlet A's data goes on shard 1, outlet B on shard 2. Cross-outlet queries are rare and can hit a read replica.
- **Event-driven side effects.** Inventory deduction, notifications, analytics are emitted as events. When the in-process bus is replaced with Kafka, listeners become independent services that can scale separately from the order path.
- **Client-generated UUIDs.** No bottleneck on a centralized ID generator. Sharding doesn't break ID uniqueness.
- **Idempotency at the request layer.** Retries are safe. This means you can put aggressive timeouts and circuit breakers in front of the service without fear of duplicating work.

These are all decisions made in the small system that pay off when scaling. The opposite — auto-incrementing IDs, in-memory caches keyed by user, side-effects called inline from request handlers — would require painful rewrites at every scaling step.

### 10.5 What would NOT scale gracefully

It's worth being honest about what the current code would need to change for high scale:

- **No queueing of write-side events.** A payment that's slow to record blocks the request. At very high load, writes should be async via a queue, with the client polling for status. This is significant work.
- **No backpressure handling.** Under sustained overload, the system will degrade by timing out rather than gracefully shedding load. Production needs explicit rate limiting (per-tenant, per-endpoint).
- **No distributed tracing / observability.** Diagnosing latency spikes at scale requires per-request tracing (e.g., OpenTelemetry). Currently the only signal is logs and HTTP response codes.

These are operational maturity concerns rather than architectural rewrites — important, but not gating for the architecture itself.

## 11. Trade-offs Made

A summary of the explicit choices in this design, the alternative that was considered, and the reasoning.

| Decision | Alternative considered | Reasoning |
|---|---|---|
| **Single PostgreSQL database** | Microservices with database-per-service | Right architecture for this scale and timeline. A single DB gives free transactional consistency across orders, inventory, bills, payments — which would otherwise require distributed transactions or saga patterns. Microservices are an evolution path (§12), not a starting point. |
| **UUID primary keys for orders** | Auto-incrementing integers | UUIDs are client-generatable, support offline ID minting, and don't bottleneck on a central sequence at scale. Slight cost in storage and index size; well worth it. |
| **Both pessimistic and optimistic locking** | Pick one | Pessimistic is correct but blocks; optimistic is fast but requires retry logic. Using each where it fits — pessimistic for inventory and payments (high-correctness, brief locks), optimistic for order edits (low-conflict, high-throughput) — is the right balance. |
| **Idempotency keys at request layer + state guards at data layer** | Either alone | Defense in depth. Idempotency catches client-side retries; state guards catch any path that bypasses the cache. Both layers compose; neither alone is sufficient. |
| **Shared-schema multi-tenancy** | Database-per-tenant or schema-per-tenant | Cheapest to operate, simplest to query, scales to thousands of tenants. Acceptable isolation for the threat model (small-business POS, no hostile tenant boundary). |
| **In-process EventEmitter** | Kafka/RabbitMQ from day one | The application code shape is the same; swapping the transport later is straightforward. Adding a broker on day one would have been infrastructure overhead with no current benefit. |
| **Snapshot prices in `order_items.unit_price`** | Always reading current menu price | Snapshots are correct: yesterday's bill must reflect yesterday's prices, even if today's menu changed. The alternative is wrong, not slower. |
| **`CHECK` constraints at DB level for stock and status** | Trust the application code | Defense in depth. Application bugs, manual SQL by a DBA, or an unconsidered code path can all violate invariants the application thinks it enforces. The database is the last guarantee. |
| **Bills computed at generation, not on read** | Compute totals on every read | A bill is a financial record. It must be deterministic and stable from the moment it's generated, regardless of subsequent menu price changes or rounding logic updates. |
| **Inventory NOT auto-restored on cancellation** | Auto-restore | Real-world cancellation is messy (kitchen may have already used ingredients). Restocking is a deliberate manager-initiated action, not an automatic side effect. |
| **No frontend** | Build a minimal UI | Assignment said "minimal, not a focus." Time spent on UI was time not spent on architecture, idempotency, locking, and concurrency — which is what the assignment is actually evaluating. |
| **Offline sync designed but not built** | Build a partial offline implementation | A half-built offline-sync system is worse than none — it produces silent data corruption. The honest choice is to build the online core well and document the offline design thoroughly. |

## 12. Future Work and Microservices Evolution

### 12.1 Immediate next steps

These are the gaps closest to ready for implementation:

1. **Idempotency on order creation.** Same pattern as payment; ~30 minutes of code.
2. **Split payments.** Schema supports it; application logic needs to allow multiple payment rows summing to the bill total before marking the bill PAID. ~1 hour.
3. **Authentication and authorization.** JWT-based auth, with the authenticated outlet_id used in every query. PostgreSQL Row-Level Security as a second layer. ~1 day.
4. **Manager-initiated inventory restock endpoint.** Used for cancelled orders, returns, manual stock corrections. ~half a day.
5. **Idempotency-key TTL.** Either move to Redis with a 24-hour expiry, or run a daily cleanup job. ~1 hour.

### 12.2 Microservices evolution

The current system is a well-bounded monolith — it deserves to *be* a monolith at this scale, but it's structured so that splitting it later is mechanical.

The natural service boundaries follow the existing service modules:

| Service | Responsibility | Owns |
|---|---|---|
| **Order Service** | Order lifecycle, state machine | `orders`, `order_items` |
| **Inventory Service** | Stock tracking and reservations | `inventory` |
| **Billing Service** | Bill generation, taxes, discounts | `bills` |
| **Payment Service** | Payment recording, idempotency | `payments`, `idempotency_keys` |
| **Catalog Service** | Outlets, menus, tax configuration | `outlets`, `menu_items` |

The hard part of the split is that **inventory deduction is currently atomic with order confirmation in the same transaction**. Once Inventory is a separate service, this atomicity is lost and must be replaced by one of:

**Option A — Saga pattern.** Order Service issues a `ReserveInventory` command to Inventory Service. On success, Order Service marks order CONFIRMED and emits `order.confirmed`. On failure or timeout, Order Service compensates (e.g., marks the order CANCELLED). The "transaction" becomes a sequence of compensable steps coordinated by the application.

**Option B — Two-phase commit.** Coordinated commit across both services. Conceptually clean but operationally fragile; has fallen out of favor.

**Option C — Eventual consistency with reconciliation.** Order Service decrements inventory optimistically, emits `order.confirmed`. Inventory Service consumes the event and adjusts stock. If the inventory service later detects oversold state, it triggers a refund/apology workflow. This is what real payment systems often do — accept the occasional bad state and have a recovery path, rather than try to prevent it perfectly.

The right answer depends on business tolerance for occasional oversells (which is real, not hypothetical — even single-database systems have this risk during a hardware failure mid-transaction). Most production POS systems pick Option C for inventory and Option A for payments.

### 12.3 Communication between services

The event bus (currently in-process) becomes the inter-service backbone. Kafka or RabbitMQ replaces `EventEmitter`; the application-level event names (`order.confirmed`, `payment.recorded`, etc.) stay identical.

For request/response between services (e.g., Order Service asking Catalog Service for current menu prices), use HTTP/JSON or gRPC. gRPC is preferable at scale for its strong typing and streaming support, but HTTP is simpler to start with and is a fine first cut.

### 12.4 Observability prerequisites

Splitting the monolith without observability is a recipe for opaque outages. Before any service split, the prerequisites are:

- **Distributed tracing** (OpenTelemetry, Jaeger): every request gets a trace ID that propagates across services. Latency spikes can be attributed to the right service.
- **Centralized logging** (ELK, Loki, or a SaaS): structured logs aggregated across services with correlation IDs.
- **Per-service metrics** (Prometheus + Grafana): request rates, error rates, p50/p95/p99 latencies per endpoint.
- **Health checks and circuit breakers**: services should fail fast when a downstream service is unhealthy, rather than queueing requests indefinitely.

These are operational maturity concerns, not architectural ones, but they gate when a microservices split makes sense versus when it would just make outages harder to debug.
